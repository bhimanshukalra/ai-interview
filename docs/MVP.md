# MVP Definition

This document defines the bare minimum needed for the AI interview project to be considered done.

The MVP is not a full hiring platform. It is a useful practice product where a signed-in user can create an AI interview, answer questions, receive feedback, and return to past sessions.

## MVP Goal

A user can:

- [x] Create an account.
- [x] Create a mock interview.
- [x] Answer AI-generated questions.
- [x] Save progress.
- [x] Generate an AI feedback report.
- [x] Return later to resume an interview or view a report.

If the product does those reliably, it is MVP-complete.

## Must Have

### Authentication

- [x] User can register.
- [x] User can log in.
- [x] User session persists in the browser.
- [x] User can log out.
- [x] Interview data is scoped to the signed-in user.

### Interview Creation

- [x] User can create an interview from `/interviews/new`.
- [x] User can choose:
  - role
  - level
  - interview type
  - optional topic
  - question count
- [x] API validates create-interview input.
- [x] Questions are generated by the configured AI provider, with mock fallback where allowed.

### Interview History

- [x] Signed-in home page shows previous interviews.
- [x] Each history item shows:
  - role
  - level
  - type
  - topic when present
  - status
  - answered count
  - score when report exists
- [x] User can resume an interview.
- [x] User can view a completed report.

### Interview Session

- [x] User can load an interview by ID.
- [x] User can answer each question.
- [x] User can save answers.
- [x] Saved answers are restored when returning to the session.
- [x] User can move forward and backward between questions.
- [x] User can only generate a report after all questions are answered.

### AI Evaluation Report

- [x] User can generate a report after completing all questions.
- [x] Report includes:
  - overall score
  - per-question score
  - answer summary
  - strengths
  - improvements
  - follow-up question when available
- [x] User can return from report to interview.

### Error and Loading States

- [x] Auth screens handle loading and errors.
- [x] Dashboard handles loading, empty, error, and loaded states.
- [x] Interview session handles loading and errors.
- [x] Report screen handles loading and errors.
- [x] API errors are safe to display and do not leak internals.

### Deployment Readiness

- [x] Web app can be deployed to Vercel.
- [x] API can be deployed to Cloudflare Workers.
- [x] Required environment variables are documented.
- [x] Database migrations exist for required tables.
- [x] Production AI provider settings are documented.

## Must Pass Before Calling MVP Done

Run and confirm:

- [ ] `pnpm -r typecheck`
- [ ] `pnpm --filter web build`

Manual smoke test:

- [ ] Register a new user.
- [ ] Create an interview.
- [ ] Answer every question.
- [ ] Generate a report.
- [ ] Go back to the dashboard.
- [ ] Confirm the interview appears with score/status.
- [ ] Log out.
- [ ] Log back in.
- [ ] Resume the same interview.
- [ ] View the report again.

## Nice To Have, Not Required For MVP

These are valuable, but not required to call the MVP done:

- [ ] WebRTC video calls
- [ ] Collaborative code editor
- [ ] Single-user code editor
- [ ] Code execution
- [ ] Report export
- [ ] Reviewer/admin views
- [ ] Adaptive follow-up questions
- [ ] Study plan generation
- [ ] Interview sharing links
- [ ] Dashboard filters/search
- [ ] Password reset
- [ ] Email verification
- [ ] Automated end-to-end tests

## Current MVP Status

Core MVP is mostly complete if the following already work:

- [x] Authentication
- [x] Protected interview data
- [x] Interview creation
- [x] Answer submission
- [x] Report generation
- [x] Interview history dashboard

Remaining MVP hardening should focus on:

- [ ] Manual smoke testing the full flow
- [ ] Tightening loading/error/empty states
- [ ] Confirming production environment setup
- [ ] Documenting known limitations

## Definition Of Done

The project can be called MVP done when:

- [ ] A real user can complete the full create-answer-report-return flow without developer help.
- [ ] The app survives refresh/login/logout across the same flow.
- [x] User-owned interview data is protected.
- [ ] The deployed web and API environments are configured.
- [ ] The smoke test passes.
- [ ] Known limitations are documented.
